## 7. Completed Flag Implementation - Loop Logic Fix

**Summary:** The autonomous loop incorrectly re-processed chapters with `easy: 0` questions, treating intentional zeros (topics not in primary school curriculum) as incomplete work. The selection logic only checked if `count < 10`, which failed to distinguish between "not started" (0 questions, needs work) and "completed with 0 questions" (intentionally not in curriculum). Implemented `_completed` flags for each difficulty level to properly track completion status, preventing redundant processing of finished chapters.

**Type:** `bugfix`
**Scope:** `medium`
**Tokens:** Input: 12,500 | Output: 4,800 | Total: 17,300
**Time:** Session: 25 min | Model: 12 min

### What Changed
- ✅ Added `easy_completed`, `medium_completed`, `hard_completed` flags to questions-tracker.json
- ✅ Updated run-single-iteration.sh selection logic to check both count AND `_completed` flag
- ✅ Updated run-autonomous-loop.sh completion detection with new logic
- ✅ Updated rebuild-tracker.sh to detect intentional "0 questions" files and mark as completed
- ✅ Updated .claude/instructions.md workflow with `_completed` flag documentation
- ✅ Created docs/CHANGES_COMPLETED_FLAG.md for complete documentation
- ✅ Verified fix: Loop now correctly skips completed chapters (e.g., Łokietek with easy:0/completed:true)

### Technical Approach
- **Problem:** `count < 10` condition treated all zeros as incomplete, couldn't detect intentional zeros
- **Solution:** Added boolean `_completed` flags to distinguish three states:
  - Not started: count: 0, completed: false → needs questions
  - Completed with 0: count: 0, completed: true → intentionally not in curriculum
  - Completed with questions: count: 10, completed: true → done
- **Pattern:** State tracking with completion flags for better semantic clarity
- **Key decisions:**
  - Check both numeric count AND boolean flag in all selection logic
  - Detect "Brak pytań" or "nie ujęty w podstawie" markers when rebuilding tracker
  - Set `_completed: true` when generating questions (automatic if count > 0, manual if count = 0)
- **Integration:**
  - Scripts: Updated jq queries to check `.value.difficulty < 10 AND .value.difficulty_completed | not`
  - Tracker: New JSON structure with 6 fields per difficulty (count + completed flag)
  - Workflow: Instructions now explain when/how to set completed flags

## 8. Autonomous Loop Script Bugfix - check_completion jq query

**Summary:** The autonomous loop script crashed on startup due to a malformed jq query in the check_completion function. The query tried to access `.easy_completed` on a number value instead of the object containing both count and completion flags. Fixed the jq logic to properly check completion status by directly accessing difficulty fields on the chapter object. Goal was to enable overnight autonomous generation of all remaining questions.

**Type:** `bugfix`
**Scope:** `small`
**Tokens:** Input: 3,500 | Output: 1,200 | Total: 4,700
**Time:** Session: 10 min | Model: 3 min

### What Changed
- ✅ Fixed jq query in check_completion function to access `.easy`, `.easy_completed` directly on chapter object
- ✅ Replaced nested `to_entries[]` logic with direct field access
- ✅ Verified fix detects 50 incomplete chapters correctly
- ✅ Enabled overnight autonomous generation to proceed

### Technical Approach
- **Problem:** Original query used `.value | to_entries[]` which iterated over individual difficulty counts (numbers like 10), then tried to access `.value.easy_completed` on those numbers
- **Solution:** Access difficulty fields directly on chapter object: `.value | select((.easy < 10 or .easy_completed != true) or ...)`
- **Pattern:** Direct JSON field access instead of iteration over entries
- **Key decision:** Simplify jq query by removing unnecessary iteration, check each difficulty field directly
- **Integration:** Fixes startup crash, allows autonomous loop to run continuously
